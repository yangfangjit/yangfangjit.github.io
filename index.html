Java Exception

What Is an Exception?

The term exception is shorthand for the phrase “exceptional event.”

Definition: An exception is an event, which occurs during the execution of a program, that disrupts the normal flow of the program’s instructions.
定义：异常是一个发生在程序执行期间的事件，它破坏了程序指令的正常流程。

When an error occurs within a method, the method creates an object and hands it off to the runtime system. The object, called an exception object, contains information about the error, including its type and the state of the program when the error occurred. Creating an exception object and handing it to the runtime system is called throwing an exception.
当方法中出现错误时，方法将会创建一个对象，然后把它交给运行时系统。这个对象被称之为异常对象，它携带了关于这个错误的信息，包括错误类型以及程序出错时的状态。创建异常对象并把它交给运行时系统就叫做抛出异常。

After a method throws an exception, the runtime system attempts to find something to handle it. The set of possible “somethings” to handle the exception is the ordered list of methods that had been called to get to the method where the error occurred. The list of methods is known as the call stack (see the next figure).
方法抛出异常后，运行时系统将尝试去找到“something”来处理这个异常。一系列可能的处理异常的“something”就是一个有序的、已被调用的方法列表，方法列表终止于错误出现处。这个方法列表通常被熟知为调用栈。

The runtime system searches the call stack for a method that contains a block of code that can handle the exception. This block of code is called an exception handler. The search begins with the method in which the error occurred and proceeds through the call stack in the reverse order in which the methods were called. When an appropriate handler is found, the runtime system passes the exception to the handler. An exception handler is considered appropriate if the type of the exception object thrown matches the type that can be handled by the handler.
运行时系统检索调用栈来寻找一个含有处理该异常的代码块的方法，该代码块也被称之为异常处理器。检索过程开始于错误发生的地方，反向搜索整个调用栈，当找到对应的处理器时，运行时系统将会把异常传递给该处理器。处理器被认为是对应的意味着抛出的异常类型匹配了该处理器所能处理的异常类型。

The exception handler chosen is said to catch the exception. If the runtime system exhaustively searches all the methods on the call stack without finding an appropriate exception handler, as shown in the next figure, the runtime system (and, consequently, the program) terminates.
异常处理器选择也被叫做捕获异常，如果运行时系统搜尽所有调用栈方法却没有找到合适的异常处理器，如下图所示，运行时系统（程序）将会终止。

如何来解释版本管理的概念呢？

栗子1，我相信大部分人是写过毕业论文的，论文写作是一个较为漫长的过程：创建文档V0版、写点东西、再写点东西、画个图、再写点、删删减减、哎呀备个份V1版、继续写、给老师看看V2版、老师批注一个字——改、改一改V3版、再该老师看看、接着改…

假设我们没有备份，但是我们想看看或者改回原来的版本时，我们能怎么做？

除了展开想象的翅膀，别无他法。

栗子2，项目经理带着我们三个开发一起写文档，我们分别在各自的机器上进行文档编写，因为是临时性工作，我们没有使用版本管理工具。我们各自编写各自负责的模块，尽可能不修改同一模块。我们先在服务器的一个文件夹中建立了一个初始文档，调好格式，命名为XXXv0.docx，然后四人各自拷贝一份至自己的电脑上开始编写各自的模块，约定编写完成后更名为XXXv1.docx（版本号加1），然后拷贝至服务器，这时会出现两种情况：1、出现命名冲突，说明已经有人抢先提交了文档，那么我们就要将版本号最大的文档拷贝至自己的电脑中，然后合并入自己修改的模块，将版本号再次加1然后拷贝会服务器公共文件夹（如果冲突那就需要再次拷贝最新的文档至本地目录合并入自己的修改，如此往复）；2、没有冲突就说明当前拷回的版本已经是最新的。这里面其实就暗含版本管理的思想，当然版本管理工具，比如Git，就远比这复杂。

作为计算机专业的学生，我很惭愧，在写论文的时候没有使用SVN或者Git作为版本管理工具，而是使用的百度云盘，每天将最新的论文稿上传至云盘中。

Git的三种状态

Git有三种状态，我们编写的文件可能处于其中之一。理解这三种状态对于Git的使用起到至关重要的作用。

已提交(committed)：已提交表示数据已经安全的，保存在本地数据库中；

已修改(modified)和：已修改表示修改了文件，但还没保存到数据库中；

已暂存(staged)：已暂存表示对一个已修改文件的当前 版本做了标记，使之包含在下次提交的快照中。

由此引入 Git 项目的三个工作区域的概念:Git 仓库、工作目录以及暂存区域。

基本的 Git 工作流程如下:

  1. 在工作目录中修改文件。

  2. 暂存文件，将文件的快照放入暂存区域。

  3. 提交更新，找到暂存区域的文件，将快照永久性存储到 Git 仓库

git commit -a可以跳过”繁琐的”git add步骤，当然了，个人认为修改一点就git add一点，也比较好注释这段提交，笼统地一次性提交就说不清改了什么。
