Posted on 2018-04-03

如何来解释版本管理的概念呢？

栗子1，我相信大部分人是写过毕业论文的，论文写作是一个较为漫长的过程：创建文档V0版、写点东西、再写点东西、画个图、再写点、删删减减、哎呀备个份V1版、继续写、给老师看看V2版、老师批注一个字——改、改一改V3版、再该老师看看、接着改…

假设我们没有备份，但是我们想看看或者改回原来的版本时，我们能怎么做？

除了展开想象的翅膀，别无他法。

栗子2，项目经理带着我们三个开发一起写文档，我们分别在各自的机器上进行文档编写，因为是临时性工作，我们没有使用版本管理工具。我们各自编写各自负责的模块，尽可能不修改同一模块。我们先在服务器的一个文件夹中建立了一个初始文档，调好格式，命名为XXXv0.docx，然后四人各自拷贝一份至自己的电脑上开始编写各自的模块，约定编写完成后更名为XXXv1.docx（版本号加1），然后拷贝至服务器，这时会出现两种情况：1、出现命名冲突，说明已经有人抢先提交了文档，那么我们就要将版本号最大的文档拷贝至自己的电脑中，然后合并入自己修改的模块，将版本号再次加1然后拷贝会服务器公共文件夹（如果冲突那就需要再次拷贝最新的文档至本地目录合并入自己的修改，如此往复）；2、没有冲突就说明当前拷回的版本已经是最新的。这里面其实就暗含版本管理的思想，当然版本管理工具，比如Git，就远比这复杂。

作为计算机专业的学生，我很惭愧，在写论文的时候没有使用SVN或者Git作为版本管理工具，而是使用的百度云盘，每天将最新的论文稿上传至云盘中。

Git的三种状态

Git有三种状态，我们编写的文件可能处于其中之一。理解这三种状态对于Git的使用起到至关重要的作用。

已提交(committed)：已提交表示数据已经安全的，保存在本地数据库中；

已修改(modified)和：已修改表示修改了文件，但还没保存到数据库中；

已暂存(staged)：已暂存表示对一个已修改文件的当前 版本做了标记，使之包含在下次提交的快照中。

由此引入 Git 项目的三个工作区域的概念:Git 仓库、工作目录以及暂存区域。

基本的 Git 工作流程如下:

  1. 在工作目录中修改文件。

  2. 暂存文件，将文件的快照放入暂存区域。

  3. 提交更新，找到暂存区域的文件，将快照永久性存储到 Git 仓库

git commit -a可以跳过”繁琐的”git add步骤，当然了，个人认为修改一点就git add一点，也比较好注释这段提交，笼统地一次性提交就说不清改了什么。
